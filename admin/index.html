🔐 什么是 ECH？

ECH（Encrypted Client Hello，加密客户端问候） 是 TLS 的一项新特性，用来 将原本会明文暴露的域名信息一起加密。

如果你希望了解更完整、偏技术细节的说明，可以参考 Cloudflare 官方博客：
👉 https://blog.cloudflare.com/zh-cn/announcing-encrypted-client-hello/

📖 背景说明

当我们使用 WS + TLS 的节点进行代理时：

  • ✅ 实际通信内容已经被 TLS 加密，GFW 无法看到你访问的具体内容
  • ❌ 但在 TLS 握手阶段，仍然会暴露一个关键信息：SNI

SNI 就是节点的伪装域名（HOST）

也就是说：

GFW 虽然不知道你在访问什么，但知道你在"扶墙"，并且知道你连接的是哪个域名。

⚠️ 会带来什么问题？

这就是为什么在以下场景中经常出现异常情况：

  • ❌ v2rayN 批量测试真链接延迟
  • ❌ Clash 策略组自动选择节点

表现通常为：

所有节点瞬间 -1，全部无法使用

其原因并不是节点真的失效，而是 运营商 / GFW 通过阻断节点域名的访问来干扰代理连接。
由于这种阻断大多是自动化策略，容易误判，因此 过一段时间节点又可能恢复正常，反复循环。

💡 ECH 的作用

ECH 的核心作用是：

将 SNI（也就是节点域名）加密

启用 ECH 后：

  • ✅ GFW 无法获取真实的节点域名
  • ✅ 外部观察到的域名将统一显示为：cloudflare-ech.com

这从源头上 阻断了通过域名精准封锁节点的手段。

🤔 ECH 会不会被封？

理论上，GFW 只需 直接阻断 cloudflare-ech.com 即可。

但这样会：

  • ❌ 误伤大量正常使用 ECH 的网站和服务
  • ❌ 带来较高的封锁成本和副作用

因此，ECH 能持续多久，取决于 GFW 的取舍与策略，至少在目前阶段仍然非常有效。

🔍 如何确认 ECH 是否已经生效？

可以通过一个 简单且直观的方法 来验证 ECH 是否正常工作。

📋 验证步骤

  1. 打开节点的 「⚙️ 详细配置信息」
  2. 在 HOST 中手动填写一个 你当前项目的已被墙的域名，例如： *.workers.dev
  3. 保存配置后 重新更新订阅
  4. 在节点列表中查找：
    • - HOST 为 *.workers.dev 的节点
    • - 并尝试连接该节点

✅ 如何判断结果？

  • ✅ 如果节点可以正常连接使用
    说明 ECH 已成功生效，真实被墙的域名已被隐藏，对外仅显示为 cloudflare-ech.com
  • ❌ 如果节点无法连接
    说明 ECH 未生效，或客户端版本、DNS 配置存在问题

⚠️ 关闭 TLS传输层加密

🔒 你正在准备关闭 Shadowsocks 的 TLS 传输层加密。为了避免节点失联，请先确认部署环境满足要求。

☁️ 关闭 TLS 前，请确认以下三点：
  1. 项目必须部署在 Workers（不是 Pages）。
  2. 在 Cloudflare 中关闭 “始终使用 HTTPS”
    SSL/TLS > 边缘证书 > 始终使用 HTTPS > 关闭
    或 HOST 直接使用 *.workers.dev 项目分配的域名。
  3. 使用的优选IP/域名的 端口 必须是 HTTP 类型端口，例如： 80、8080、8880、2052、2082、2086、2095。

📘 传输与加密说明

📡 关闭 TLS 后，数据将以 HTTP 明文形态传输，不再具备 TLS加密 对链路的封装与伪装能力。

🔐 但节点仍由 Shadowsocks AEAD（aes-128-gcm / aes-256-gcm）加密，流量数据本身依旧是加密传输。

✅ 总结：最终流量会以 明文的方式 传输 加密数据。
